## Naming Note (Important)

The repo is called `web`. Inside it, `apps/web` is the **landing page** (riplai.id).  
These are two different things:

| | What it is |
|---|---|
| `web/` (repo root) | The entire Turborepo monorepo |
| `web/apps/web` | The public-facing landing page app |
| `web/apps/merchant` | The merchant PWA (inbox + dashboard) |
| `web/apps/admin` | The admin panel |

## Before You Start — Cross-Repo Contracts

### What you need from `api` before building each section

| Web section | Needs from api | Can build without? |
|---|---|---|
| Landing page | Nothing | Yes — build now |
| Auth (login page) | `POST /auth/login` shape | Yes — use fixture |
| Inbox list | `GET /conversations` shape | Yes — use fixture |
| Conversation thread | `GET /conversations/:id` shape | Yes — use fixture |
| Reply form | `POST /conversations/:id/reply` | Yes — use fixture |
| SSE (real-time) | `GET /sse/events` running | No — need api Phase 10 |
| Dashboard stats | `GET /dashboard/stats` | Yes — use fixture |

### Mock strategy (build ahead of api)
No MSW needed — keep it simple. Create `packages/api-client/src/__fixtures__/` with hardcoded JSON responses matching the expected api shapes. Swap `api-client/src/client.ts` to real Axios once api is ready.

```typescript
// packages/api-client/src/client.ts
// Toggle this constant while api is still being built:
const USE_FIXTURES = process.env.NEXT_PUBLIC_USE_FIXTURES === 'true'
```

### Type sharing
You write Go structs first in `api`. Mirror them here manually:
- Go `ConversationResponse` → `web/packages/types/src/api/conversation.ts`
- Go `MessageResponse` → `web/packages/types/src/api/message.ts`
- Go `MerchantResponse` → `web/packages/types/src/api/merchant.ts`

Do this **at the start of each api phase** that adds new endpoints — not at the end.

> Needs: `POST /auth/login` shape from api (or fixture)

- [ ] `app/layout.tsx` — global layout
- [ ] `app/(auth)/login/page.tsx` — login form
- [ ] `app/(auth)/login/LoginForm.tsx` — email + password, calls `/auth/login`
- [ ] `store/auth.store.ts` (Zustand) — `token`, `merchant`, `setAuth()`, `logout()`
- [ ] `app/(app)/layout.tsx` — protected layout: redirects to /login if no token

### JWT handling
- [ ] Store token in `localStorage` (key: `riplai_token`)
- [ ] Axios interceptor in `api-client` reads it and attaches `Authorization: Bearer` header
- [ ] On 401 response → clear token + redirect to `/login`

**Exit condition:** Login works (against fixture or real api). Protected routes redirect to login without token.

## Phase 4 — Merchant App: Inbox (F5)
> Core feature. Mobile-first PWA. This is what merchants use every day.  
> Needs: conversations + messages API shapes (or fixtures), SSE (or polling fallback)

### Layout
```
┌─────────────────────────────────┐
│ Header: Dewi Inbox  [●] open    │
├──────────────┬──────────────────┤
│ Conversation │ Message thread   │
│ List         │                  │
│              │ [customer msg]   │
│ ● Needs reply│   [bot reply]    │
│ ○ Bot handled│                  │
│ ○ Waiting    │ [Reply input]    │
└──────────────┴──────────────────┘
```

### Conversation list (`components/inbox/`)
- [ ] `ConversationList.tsx` — scrollable list, sorted by `last_message_at`
- [ ] `ConversationItem.tsx` — customer number + preview + `StatusBadge`
- [ ] `StatusBadge.tsx` — `🔴 Needs Reply` / `✅ Bot Handled` / `⏳ Waiting`
- [ ] `hooks/useConversations.ts` (TanStack Query) — `GET /conversations`, auto-refetch 30s

### Thread view
- [ ] `app/(app)/inbox/page.tsx` — list view
- [ ] `app/(app)/inbox/[id]/page.tsx` — thread view
- [ ] `components/inbox/MessageBubble.tsx` — left (customer) vs right (bot/owner), timestamp
- [ ] `components/inbox/ReplyForm.tsx` — textarea + send button, calls `POST /conversations/:id/reply`
- [ ] `hooks/useConversation.ts` (TanStack Query) — `GET /conversations/:id`
- [ ] `hooks/useSendReply.ts` (TanStack mutation) — POST + invalidate conversation query
- [ ] `store/inbox.store.ts` (Zustand) — `activeConversationId`, `replyDraft`

### Real-time (SSE)
- [ ] `hooks/useSSE.ts` — wraps `EventSource`, reconnects on drop
  - On `new_message` event → invalidate `conversations` query (TanStack does the rest)
  - On `status_change` event → invalidate affected conversation
- [ ] Fallback: if SSE not yet ready in api, polling every 10s is fine temporarily

**Exit condition:** Merchant can see all conversations, open a thread, read the history, type a reply, and send it. Status badges update correctly.
